# ADR-0002: Redis for Reservation Hot Path

- Status: Accepted
- Date: 2026-03-25

## Context

티켓 예매 도메인은 특정 시점에 요청이 급격히 몰리는 특성이 있다. 저장소에는 Spring Data Redis 설정과
`RedisService` 가 공통 인프라로 존재하며, README의 서비스 소개 역시 트래픽 집중 상황을 전제로 한다.

예매 시나리오에서 모든 요청을 즉시 관계형 데이터베이스만으로 처리하면 좌석 점유 경쟁, 병목, 응답 지연이 커질 수 있다.

## Decision

예매의 고트래픽 구간에서는 Redis를 보조 저장소이자 빠른 접근 계층으로 사용한다.

## Consequences

### Positive

- 좌석 점유와 같은 짧은 생명주기의 상태를 빠르게 처리할 수 있다.
- 예매 집중 시간대의 데이터베이스 부담을 줄일 수 있다.
- 여러 애플리케이션 인스턴스가 동일한 인메모리 상태를 공유할 수 있다.

### Negative

- 로컬 실행 시 MySQL 외에 Redis도 함께 준비해야 한다.
- 테스트 작성 시 Redis를 목 처리하거나 별도 테스트 인프라를 구성해야 한다.
- 만료 정책, 정합성, 장애 복구 전략을 추가로 관리해야 한다.

## Notes

테스트나 문서에는 Redis 의존성을 명시적으로 적어 신규 개발자가 실행 조건을 바로 이해할 수 있게 한다.
